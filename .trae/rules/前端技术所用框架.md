# 前端技术所用框架规则

> 本规则适用于本仓库所有前端项目，AI 与人工开发均须遵守。规则按优先级从高到低排列，冲突时以靠前者为准。

---

## 一、技术栈基线

1. **UI 框架固定为 Bootstrap 5.3**
   - 所有样式优先使用 Bootstrap 5.3 内置的 utility class 与组件（grid、flex、spacing、卡片、模态框、表单、按钮等）。
   - 禁止引入 jQuery、Bootstrap 4 或其它 CSS 框架（Tailwind / Bulma / Foundation 等）。
   - 如需图标，统一使用 Bootstrap Icons。
   - 自定义 CSS 仅在 Bootstrap 无法满足时编写，且必须以 CSS 变量形式集中管理（见第六条）。

2. **JS 规范**
   - 优先使用原生 ES6+ 语法；如项目引入了框架（如 Vue / React），以项目实际配置为准，但样式层仍必须用 Bootstrap 5.3。
   - 禁止使用 `var`，统一用 `const` / `let`。
   - 禁止使用 `any`（TS 项目中），必须显式标注类型。

---

## 二、响应式设计（强制）

1. **所有页面与组件必须响应式**，确保手机、平板、桌面、超大屏均可正常访问与操作。
2. **布局优先使用 Bootstrap 栅格系统**（`container` / `row` / `col-*`），不要手写媒体查询除非必要。
3. 断点遵循 Bootstrap 默认：`xs <576px`、`sm ≥576px`、`md ≥768px`、`lg ≥992px`、`xl ≥1200px`、`xxl ≥1400px`。
4. 图片必须使用 `img-fluid`，表格在小屏使用 `table-responsive`，表单控件使用 Bootstrap 的 `form-control` / `form-select` 默认响应式行为。
5. 交互元素（按钮、链接、表单）的触摸目标在移动端不小于 44×44px。
6. 禁止出现横向滚动条；任何固定宽度必须配合断点回退到百分比或 `auto`。

---

## 三、命名规范

1. **语义化优先**：命名要表达"是什么/做什么"，而非"长什么样"。
   - ✅ `userAvatar`、`handleSubmit`、`orderCard`、`paymentStatusBadge`
   - ❌ `div1`、`redBox`、`temp`、`data2`、`btnClick`
2. **大小写约定**
   - 变量、函数、属性：`camelCase`（如 `fetchUserList`）。
   - 类、组件、构造函数：`PascalCase`（如 `UserProfile`、`OrderCard`）。
   - 常量（不可变配置）：`UPPER_SNAKE_CASE`（如 `MAX_RETRY_COUNT`、`API_BASE_URL`）。
   - CSS 类名（自定义部分）：`kebab-case` 且带项目前缀（如 `chat--message-bubble`）。
   - 布尔变量以 `is` / `has` / `should` 开头（如 `isLoading`、`hasError`、`shouldRetry`）。
3. **事件处理函数**以 `handle` 或 `on` 开头（如 `handleLoginClick`、`onMessageReceived`）。
4. **数据获取函数**以 `fetch` / `get` / `load` 开头；**数据提交函数**以 `save` / `submit` / `update` / `create` / `delete` 开头。

---

## 四、注释规范（强制中文）

1. **每一个函数都必须有中文注释**，使用 JSDoc 格式：
   ```javascript
   /**
    * 根据用户 ID 获取用户详细信息
    * @param {number} userId - 用户唯一标识
    * @returns {Promise<Object>} 用户信息对象，失败时抛出 Error
    */
   async function fetchUserDetail(userId) { ... }
   ```
2. 注释说明 **"为什么这样做"**，而非"做了什么"——代码本身已说明做什么。
3. 超出业务直觉的隐含约束、第三方 API 的怪异行为、性能 hack 必须注释。
4. 文件顶部用简短中文说明该文件职责（1-2 行）。
5. 禁止写无意义的注释（如 `// 循环`、`// 变量 i`）。

---

## 五、DRY 原则（不要重复自己）

1. **能定义一次的变量、常量、配置，绝对不能重复声明多次**。
   - 重复出现的字符串、数值、配置项必须抽成常量，放在文件顶部或统一的 `config.js` / `constants.js` 中。
   - 跨文件复用的常量集中在 `src/constants/` 目录。
2. **能用循环解决的，禁止复制粘贴**。
   - 连续 3 次以上结构相似的代码块，必须用 `for` / `forEach` / `map` / `reduce` 重构。
   - 同一段逻辑出现两次以上，抽成函数。
3. **配置数据与渲染逻辑分离**：菜单、选项、状态映射等用数组/对象维护，配合循环渲染，禁止手写一份份重复的 DOM/JSX。
4. **工具函数集中管理**：`src/utils/` 下按职责分文件（`format.js`、`validate.js`、`request.js` 等），禁止散落在各组件内部。

---

## 六、CSS / 样式规范

1. **Bootstrap utility 优先**：能用 `d-flex`、`gap-2`、`mt-3`、`text-truncate` 解决的，禁止写自定义 CSS。
2. **CSS 变量集中管理**：项目级颜色、间距、字号在 `:root` 中定义一次，业务样式引用变量：
   ```css
   :root {
     --brand-primary: #6f42c1;
     --brand-radius: 0.5rem;
     --chat-bubble-bg: #f8f9fa;
   }
   .chat--bubble { background: var(--chat-bubble-bg); border-radius: var(--brand-radius); }
   ```
3. **禁止硬编码颜色、间距**：`color: #6f42c1` 出现两次以上必须转成变量。
4. 自定义类名使用项目前缀 + BEM 思路（`chat__bubble--active`），避免与 Bootstrap 冲突。
5. 禁止使用 `!important`，除非覆盖第三方库且无其它方案（需注释原因）。

---

## 七、性能要求

1. **图片懒加载**：非首屏图片必须 `loading="lazy"`，并提供 `width` / `height` 防止布局抖动。
2. **事件委托**：列表中大量相同交互元素使用事件委托，不要逐个 `addEventListener`。
3. **避免无谓重渲染**（框架项目）：合理使用 `key`、`useMemo` / `computed`，列表 `key` 必须用稳定 ID，禁止用 index。
4. **大列表虚拟化**：超过 100 项的长列表使用虚拟滚动。
5. **防抖节流**：输入搜索、窗口 resize、滚动监听必须做 debounce/throttle（250-300ms）。
6. **资源按需加载**：路由级组件使用动态 `import()` 懒加载；Bootstrap 按需引入所需模块。
7. 禁止在循环内做 DOM 查询、网络请求、重复计算。

---

## 八、代码整洁度

1. **单一职责**：一个函数只做一件事，超过 80 行考虑拆分。
2. **提前 return**：用卫语句替代深层嵌套 `if/else`：
   ```javascript
   function processOrder(order) {
     if (!order) return;              // 卫语句
     if (!order.items?.length) return;
     // 主逻辑...
   }
   ```
3. **默认参数**：使用 ES6 默认参数，禁止用 `||` 兜底布尔值（`||` 会把 `0`、`''` 也当 falsy）。
4. **解构赋值**：对象/数组取值优先解构，减少临时变量。
5. **模板字符串**：字符串拼接一律用反引号，禁止 `+` 拼接。
6. **可选链与空值合并**：`user?.profile?.name ?? '匿名'`。
7. **禁止 console 留存在提交代码中**（调试时用，提交前移除或用统一 logger）。
8. **不写防御性代码**：内部可信调用不要做参数类型检查；仅在外部边界（用户输入、API 响应、文件读取）做校验。

---

## 九、可访问性（a11y）

1. 所有交互元素必须可键盘操作，`:focus` 样式可见。
2. 图片需 `alt`，装饰性图片用 `alt=""`。
3. 表单控件必须关联 `<label>`。
4. 颜色对比度满足 WCAG AA（正文 ≥4.5:1）。

---

## 十、AI 开发额外约束

1. **先理解再写**：动代码前通读相关文件，遵循现有风格，不要"顺手"重构。
2. **最小改动**：只触碰必须修改的部分，不扩张到相邻代码。
3. **不引入未要求的依赖**：用户没要求的库不装，新依赖必须在 PR 中说明用途。
4. **不写半成品**：要么实现完整，要么不写；占位符 `// TODO` 必须同步给出原因与后续计划。
5. **每次修改后自检**：变量是否重复声明？是否有可循环替代的重复代码？注释是否齐全？响应式是否覆盖移动端？
